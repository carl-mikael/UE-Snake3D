# SetEnv Error - Permanent Fix Applied ✅

## Problem
The build was failing with:
```
error MSB4018: The "SetEnv" task failed unexpectedly.
System.ArgumentException: Environment variable name or value is too long.
```

## Root Cause
- Unreal Engine auto-generates `Snake3D.vcxproj` with 100+ engine module include paths
- Combined path length exceeded Windows' 32KB environment variable limit
- MSBuild's `SetEnv` task tried to set the `INCLUDE` environment variable and failed

## Solution Applied
Created `Directory.Build.props` at the project root (`S:\UnrealProjects\Snake\`)

This file:
- ✅ Overrides the `SetBuildDefaultEnvironmentVariables` target for NMake projects
- ✅ Prevents the `SetEnv` task from executing (unnecessary for Unreal's build scripts)
- ✅ **Survives project regeneration** (not regenerated by Unreal Engine)
- ✅ **Auto-imported by MSBuild** (no manual configuration needed)
- ✅ **Team-wide benefit** (committed to git, all developers get the fix)

## How It Works

MSBuild automatically imports `Directory.Build.props` before processing any `.vcxproj` files.

When a Makefile/NMake project (like Unreal's Snake3D project) is built:
1. `Directory.Build.props` overrides `SetBuildDefaultEnvironmentVariables` target
2. The target becomes empty (no SetEnv calls)
3. The actual build uses Unreal's build scripts (which handle paths correctly)
4. No environment variable size issues

## Why This Is Permanent

| Aspect | Details |
|--------|---------|
| **Not regenerated** | `Directory.Build.props` is MSBuild infrastructure, not touched by Unreal's ProjectFileGenerator |
| **Auto-imported** | MSBuild automatically imports this before any `.vcxproj` files |
| **Only affects NMake** | The condition `ConfigurationType == 'Makefile'` ensures safety |
| **Survives regeneration** | When `GenerateProjectFiles.bat` runs, it won't touch this file |
| **Survives git clean** | The file is committed to git and checked out with the repo |
